## 🚨 1. Git 충돌 에러 (Conflict Error)

### 📌 1-1. push 오류 발생

![push 오류](/image/image.png)

에러 메시지 핵심:

```bash
error: failed to push some refs to
```

### ✔ 원인

- 원격 저장소가 로컬보다 최신 상태일 때
- 팀원이 먼저 push를 진행했을 때

👉 원격 저장소에 내가 가지고 있지 않은 변경 사항이 존재할 경우 발생합니다.

![오류 과정](/image/image-1.png)

---

### 🔧 1-2. 해결 방법

#### ✔ Step 1. pull

```bash
git pull origin
```

![pull 과정](/image/image-2.png)

#### ✔ Step 2. push

```bash
git push origin
```

![push 성공](/image/image-3.png)

#### 📌 기본 작업 순서

```text
add → commit → pull → push
```

---

### ⚠ 1-3. 같은 파일, 같은 위치 수정 시 발생하는 충돌

![충돌 에러](/image/image-4.png)

```bash
<<<<<<< HEAD
내 코드
=======
팀원 코드
>>>>>>> branch-name
```

![충돌 표시 화면](/image/image-5.png)

#### 🛠 해결 방법

1. 구분자 삭제
2. 최종 코드만 남김

```bash
git add .
git commit -m "conflict 해결"
git push origin
```

📌 대부분 소통 부족에서 발생 → 회의 후 수정하는 것이 바람직합니다.

## ✏ 2. 오타 (Typo)

![오타 예시](/image/image-6.png)

코딩 중 사소한 오타 하나로도:

- 코드가 실행되지 않거나
- 오류가 발생할 수 있습니다.

### ✔ 예방 방법

- 코드 천천히 다시 읽기
- 팀원 코드 리뷰
- push 전 명령어 재확인

> 작은 실수도 협업에서는 큰 영향을 줄 수 있음을 배웠습니다.

## 🤝 우리는 이렇게 소통했다

이번 프로젝트에서 우리 팀은 단순히 역할을 나누는 방식이 아니라,
질의응답과 실시간 공유를 통해 함께 학습하는 방식으로 협업을 진행하였다.

### 💬 1. 질의응답을 통한 개념 정리

- Git 명령어가 헷갈리는 부분을 서로 질문하며 정리하였다.
- clone과 pull의 차이를 다시 설명하며 개념을 명확히 하였다.
- CLI 환경 사용법을 함께 확인하며 익숙해지는 과정을 거쳤다.
- 오류 메시지를 공유하고 원인을 함께 분석하였다.

### 🛠 2. Git 오류는 함께 해결하였다

- push 오류 발생 시 개인이 해결하려 하지 않고 먼저 공유하였다.
- add 및 commit 누락 여부를 함께 점검하였다.
- origin 오타를 발견하고 수정하였다.
- remote 연결 상태를 확인하며 해결 방법을 찾았다.

## 🎯 프로젝트를 통해 배운 점

- 협업에서는 기술보다 **원활한 소통과 배려하는 태도가 더욱 중요하다는 깨달음이다.**
- 혼자 아는 것과 팀으로 직접 경험하는 것은 **전혀 다른 차원의 배움이라는 점이다.**
- Git을 이론으로 이해하는 것과 CLI 환경에서 직접 사용하는 것은 **분명한 차이가 있다는 사실이다.**
- push 전 pull을 확인하는 습관은 **올바른 버전 관리를 위한 기본이라는 점이다.**
- 충돌(conflict)은 피해야 할 문제가 아니라 **이해하고 해결할 수 있어야 할 과정이라는 배움이다.**
- 작은 오타와 사소한 실수도 전체 작업에 영향을 줄 수 있다는 것은 **꼼꼼함의 중요성을 보여주는 부분이다.**
- 코딩 전 구조를 설계하고 계획을 세우는 과정은 **작업 효율과 완성도를 높이는 핵심 단계이다.**
- 모르는 것을 질문하고 공유하는 태도는 **성장을 앞당기는 가장 좋은 방법이라는 깨달음이다.**
- 도움을 주고받는 협업은 부담이 아니라 **함께 성장하는 과정이라는 경험이다.**
- 이번 프로젝트는 어제보다 더 나아진 나 자신을 느끼게 해준 **의미 있는 성장의 시간이다.**
